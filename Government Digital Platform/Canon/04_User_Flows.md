## 1. מטרה ושיטת קריאה

המסמך מפרק את מסעות הליבה של שני הקהלים לצעדים, החלטות ומצבי קצה. כל זרימה נכתבת בפורמט אחיד:

**שחקן · מטרה · טריגר · תנאי קדם → מסלול ראשי (ממוספר) → הסתעפויות ומצבי קצה → מצב סיום + מעבר סטטוס → רכיבים.**

מוסכמות סימון: `→` צעד הבא · `◆` נקודת החלטה · `⚠` מצב קצה/שגיאה · `[Status: ...]` קטגוריית סטטוס גלובלית (D-009) · הפניות לישויות ב-CamelCase.

### 1.1 שתי הכרעות שננקטו כברירת מחדל (לאישור)

נדרשו לשם בניית הזרימות; מסומנות ⬦ לאישורך:

- ⬦ **מודל Queue (Operations):** היברידי — בקשה נכנסת לתור משותף לפי **מערכת + תפקיד**, עם אפשרות **הקצאה/שיוך ידני** ע"י מנהל תחום. (IA §8.2)
- ⬦ **עומק Launcher (Citizen):** הדרגתי — בוחרים **מערכת**; אם למערכת שירות/סוג-בקשה יחיד, מדלגים ישר אליו; אחרת מציגים רשימת שירותים. (IA §8.4, נסמך על D-008)

## 3. מסעות אזרח (Citizen Portal)

### CF-1 — כניסה והזדהות ← אזור אישי
**שחקן:** מגיש · **מטרה:** להיכנס ולראות מה דורש ממנו פעולה · **טריגר:** כניסה לפורטל · **תנאי קדם:** זהות לאומית.

1. נחיתה על מסך הזדהות אחד ונקי → פעולה ראשית בולטת "הזדהות לאומית".
2. ◆ זיהוי ראשוני? → אם אין משתמש קיים, נוצר פרופיל מפרטי ההזדהות (שם, ת.ז., מייל — לקריאה בלבד).
3. → הגעה ל**אזור אישי** (לא ל-Launcher): משימות פתוחות, סטטיסטיקות לפי Status Category, הודעות.
4. ◆ יש בקשה ב-`[Status: ממתין למגיש]`? → מוצג **Alert Banner** "ממתין לפעולתך" עם קישור ישיר (ראו XF-1).

⚠ **קצה:** הזדהות נכשלה → הודעת שגיאה מנחה + תמיכה. ⚠ אזרחות/גיל לא עומדים בסף שירות → הכוונה (לא חסימת-מודאל אטומה). ⚠ אין הרשאת גישה למערכת → הסבר *איך* משיגים גישה (לא רק "נעול").

**מצב סיום:** אזור אישי טעון. **רכיבים:** מסך הזדהות, Stats Tiles, Alert Banner, Messages Feed.

---

### CF-2 — פתיחת בקשה חדשה
**שחקן:** מגיש · **מטרה:** להתחיל בקשה לשירות · **טריגר:** "בקשה חדשה" · **תנאי קדם:** מזוהה.

1. מאזור אישי → "התחל בקשה חדשה" → **Launcher** (בחירת מערכת).
2. בחירת **System** (כרטיס). ◆ עומק (ברירת מחדל הדרגתית):
   - מערכת עם שירות יחיד → דילוג ישר ל-Request Type.
   - מערכת עם כמה שירותים → מסך בחירת **Service** → בחירת **Request Type**.
3. ◆ בדיקת זכאות/תנאי סף לסוג הבקשה → אם לא עומד, הכוונה למה חסר (ולא יצירת בקשה ריקה).
4. → נוצרת **Request** במצב `[Status: טיוטה]`, ונפתח הטופס (המשך ב-CF-3).

⚠ **קצה:** מערכת `בתחזוקה` → מצב Maintenance אחיד. ⚠ מערכת נעולה (נדרש כרטיס חכם) → הסבר דרך גישה.

**מצב סיום:** בקשה ב-`[Status: טיוטה]` + טופס פתוח. **רכיבים:** Launcher, כרטיסי System/Service, מצבי System.

---

### CF-3 — מילוי טופס, מסמכים והגשה
**שחקן:** מגיש · **מטרה:** למלא ולהגיש · **טריגר:** טופס פתוח · **תנאי קדם:** בקשה ב-`טיוטה`.

1. טופס מרובה-שלבים (Tabs/Stepper) עם Progress Bar וסימון "הושלם" (כמו M2).
2. שדות מההזדהות ממולאים מראש ולקריאה בלבד; שאר השדות ידניים.
3. ◆ שדות מותנים → נחשפים לפי תשובות (חשיפה הדרגתית).
4. **שמירה אוטומטית** מתמשכת (Toast) — אין איבוד נתונים.
5. שלב מסמכים → **העלאה מותנית**: כל מסמך נדרש מופיע לפי תשובה/מעמד (Document ↔ Field).
6. ◆ ולידציה לפני הגשה → ⚠ שדה חובה חסר / מסמך חסר → הודעת שגיאה מתחת לשדה, מנחה כיצד לתקן; ניווט ישיר לשלב החסר.
7. שלב הצהרה → אישור → **הגשה**.
8. → הבקשה עוברת ל-`[Status: הוגשה]`; נשלחת הודעת אישור; חוזרים לאזור אישי עם עדכון.

⚠ **קצה:** יציאה באמצע → הבקשה נשמרת כ-`טיוטה` ומופיעה ברשימה. ⚠ קובץ לא נתמך/גדול → שגיאת העלאה ברורה.

**מצב סיום:** `[Status: הוגשה]`. **רכיבים:** Tabs, Progress Bar, שדות + מותנים, Dropzone, Toast, Action Bar (טיוטה / בדיקה ושליחה).

---

### CF-4 — מעקב סטטוס
**שחקן:** מגיש · **מטרה:** לדעת איפה הבקשה · **טריגר:** פתיחת בקשה מהרשימה · **תנאי קדם:** בקשה קיימת.

1. "הבקשות שלי" → רשימה עם **Filter Pills** (הכל / פעילות / סגורות) ו-**Status Badge** לכל שורה (צבע=Category, טקסט=Label).
2. בחירת בקשה → מסך בקשה עם **Status Timeline** (Stage מודגש), נתונים, מסמכים, הודעות.
3. ◆ `[Status: ממתין למגיש]`? → פעולה ראשית "השלם עכשיו" בולטת (ראו XF-1).
4. ◆ `[Status: אושרה/נדחתה]`? → הצגת תוצאה והסבר; פעולות המשך אם רלוונטי.

**מצב סיום:** המגיש מבין את המצב והצעד הבא. **רכיבים:** Filter Pills, Status Badge, Status Timeline, Messages.

---

## 4. מסעות פקיד (Government Operations Platform)

### OF-1 — שולחן עבודה ← בחירת בקשה מהתור
**שחקן:** פקיד מטפל · **מטרה:** לדעת מה לטפל ולפתוח בקשה · **טריגר:** כניסת עובד · **תנאי קדם:** תפקיד מתאים.

1. כניסה → **שולחן עבודה**: התור שלי, ממתין לפעולה, חריגות SLA.
2. ◆ מקור התור (היברידי): בקשות לפי **מערכת + תפקיד**, ובנוסף בקשות **שהוקצו ידנית**.
3. סינון/מיון התור לפי **Status Category**, סוג בקשה, תאריך, מטפל.
4. בחירת בקשה → פתיחת **Case Workspace** (המשך OF-2).

⚠ **קצה:** תור ריק → Empty State מוכוון-פעולה. ⚠ בקשה שכבר נלקחה ע"י עמית → סימון/נעילת טיפול.

**מצב סיום:** Case Workspace פתוח. **רכיבים:** Workspace, Queue Tabs, Data Table (פילטרים/מיון), Empty State.

---

### OF-2 — טיפול ובדיקת בקשה
**שחקן:** פקיד מטפל / בודק מקצועי · **מטרה:** לבדוק ולקדם · **טריגר:** בקשה ב-`הוגשה/בטיפול` · **תנאי קדם:** הקצאה/הרשאה.

1. **Case Workspace**: נתוני הבקשה (Form values), מסמכים, היסטוריית סטטוס, Action Bar.
2. שינוי סטטוס ל-`[Status: בטיפול]` (Label תהליכי, למשל "בבדיקת השלמות").
3. בדיקת מסמכים → ◆ כל Document: תקין / חסר / נדחה.
4. ◆ נדרש מידע/מסמך מהמגיש? → **בקשת השלמה** (מעבר ל-XF-1) → `[Status: ממתין למגיש]`.
5. ◆ נדרשת בחינה מקצועית? → העברה ל**בודק מקצועי** (Label "בבחינה מקצועית"); הבודק מוסיף חוות דעת.
6. → כשמוכן, קידום למאשר (OF-3).

⚠ **קצה:** נתונים סותרים → סימון לבדיקה. ⚠ חוסר הרשאה לפעולה → הפעולה מוסתרת לפי Role.

**מצב סיום:** הבקשה מוכנה להכרעה או ממתינה למגיש. **רכיבים:** Case Workspace, Action Bar, בדיקת Document, Status controls.

---

### OF-3 — אישור / דחייה / העברה
**שחקן:** מאשר · **מטרה:** להכריע · **טריגר:** בקשה מוכנה · **תנאי קדם:** סמכות אישור (Role).

1. סקירת תקציר הבקשה + חוות דעת מקצועית.
2. ◆ הכרעה:
   - **אישור** → `[Status: אושרה]` (Label, למשל "נשלחה לצה"ל") → הודעה למגיש.
   - **דחייה** → `[Status: נדחתה]` + נימוק → הודעה למגיש.
   - **העברה** → לוועדה/גורם נוסף (Label "נשלחה לוועדה"), נשאר `בטיפול`.
3. → עדכון אוטומטי של ה-Citizen Portal (אותה Request, תצוגת מגיש).

⚠ **קצה:** החלטה דורשת אישור כפול → שלב אישור נוסף לפני סגירה.

**מצב סיום:** `אושרה` / `נדחתה` / `בטיפול` (הועבר). **רכיבים:** Action Bar, Status controls, Message.

## 5. XF-1 — לולאת "השלמת פרטים" (חוצת-מוצרים)

נקודת המגע המרכזית בין שני המוצרים — אותה Request, שני שחקנים.

```
[Operations] פקיד מזהה חוסר
   └─ מגדיר אילו שדות/מסמכים חסרים + הערה
   └─ Status → [ממתין למגיש]  (Label: "השלמת פרטים")
         │  טריגר תהליך
         ▼
   ── Message + Notification נשלחות למגיש ──
         │
         ▼
[Citizen] מגיש רואה:
   └─ Alert Banner באזור אישי + פעמון + Status Badge "השלמת פרטים"
   └─ "השלם עכשיו" → נפתח הטופס ממוקד בשדות/מסמכים החסרים בלבד
   └─ משלים → מגיש מחדש
         │
         ▼
   Status → [בטיפול]  → חוזר לתור הפקיד (OF-2)
```

**עקרונות:** המגיש מופנה ישירות לחוסר (לא לטופס המלא); ההערה מהפקיד מוצגת בהקשר; הסטטוס והצבע אחידים בשני הצדדים (D-009).

⚠ **קצה:** המגיש לא משלים בזמן → תזכורת אוטומטית; מדיניות מועד ⬦ לאפיון.

## 6. מפת מעברי סטטוס (Status Transitions)

מבוססת על Status Category הגלובלי (D-009). ה-Label ספציפי לתהליך.

```
טיוטה ──הגשה──▶ הוגשה ──פתיחת טיפול──▶ בטיפול
                                   │  ├─בקשת השלמה─▶ ממתין למגיש ──השלמה──▶ בטיפול
                                   │  ├─אישור──────▶ אושרה ─▶ (נסגרה)
                                   │  └─דחייה──────▶ נדחתה ─▶ (נסגרה)
טיוטה ──ביטול──▶ נסגרה
```

| מעבר | שחקן | טריגר |
|---|---|---|
| טיוטה → הוגשה | מגיש | הגשת טופס (CF-3) |
| הוגשה → בטיפול | פקיד | פתיחת טיפול (OF-2) |
| בטיפול → ממתין למגיש | פקיד | בקשת השלמה (XF-1) |
| ממתין למגיש → בטיפול | מגיש | השלמה והגשה מחדש |
| בטיפול → אושרה / נדחתה | מאשר | הכרעה (OF-3) |
| כל מצב פתוח → נסגרה | מערכת/פקיד | סגירה/ביטול |

## 7. שאלות פתוחות

1. ⬦ אישור מודל ה-Queue (היברידי) ועומק ה-Launcher (הדרגתי) — §1.1.
2. ⬦ מדיניות מועדים ל-"ממתין למגיש" (תזכורות, פקיעה).
3. ⬦ האם נדרש אישור כפול/ועדה בחלק מהתהליכים, וכיצד מיוצג ב-Timeline.
4. ⬦ האם מגיש יכול לערוך בקשה לאחר הגשה (לפני תחילת טיפול) או רק דרך "השלמת פרטים".
5. ⬦ טיפול בהגשות מטעם ארגון (נציג) — מי רואה ומי פועל.
